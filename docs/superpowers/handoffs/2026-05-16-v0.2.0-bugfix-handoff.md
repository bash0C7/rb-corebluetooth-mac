## 3. アーキテクチャ重要決定 (再議論しない)

- **3 層**: Ruby (pure routing + Data value object) → C bridge (`rb_thread_call_without_gvl` で GVL 解放) → Swift (`@c` SE-0495 ABI + CBCentralManager + delegate)
- **TypedData**: `Central` のみ `TypedData_Wrap_Struct`。Peripheral/Service/Characteristic は pure Ruby、Subscription は `Data.define(:central_id, :subscription_id)` (Ractor-shareable)
- **Notify model**: poll-style `subscription.next_value(timeout:)`、gem 内では Thread/Ractor を spawn しない
- **Concurrency**: `OSAllocatedUnfairLock<State>` (型付き lock-as-state パターン)、`@preconcurrency import CoreBluetooth`、`@unchecked Sendable` クラス。`var + 空 state lock` は Swift 6 で爆発するので使わない
- **macOS 13+** deployment (`OSAllocatedUnfairLock` 要件)
- **swift_gem** は sibling `../swift_gem` から path で参照、`bundle exec rake compile` 起動

## 4. 既知の latent issue (review で surface したが未修正)

### 高優先度 (HW テスト時に顕在化する可能性)

| # | 場所 | 内容 | レビュアー confidence |
|---|---|---|---|
| L1 | `CBMSubscriptionRegistry.purgeAll` | `for (id, entry) in dict where ...` 中で `dict.removeValue(forKey: id)`。Swift は警告出さなかったけど Dictionary iteration 中の mutation は UB。subscription 数が少ない (1-3) なら問題出ない可能性高い | 中 |
| L2 | `CBMPeripheralDelegate` (12 var) | 同一 peripheral への並行 op (例: 並行 read 2 個) は `*CharUUID` フィルタで race。設計は serial-caller assumption。caller が並行呼ぶケースで顕在化 | 70% |
| L3 | `RUBY_UBF_IO` | GCD `DispatchSemaphore.wait` も `Thread.sleep` も実は割り込めない。`Timeout::timeout` で `Central.new` / `scan` / `read` を囲っても、本体の timeout 引数が満了するまで preempt されない | 72% |

### 中優先度 (cosmetic / 文書化レベル)

| # | 場所 | 内容 |
|---|---|---|
| L4 | `corebluetooth_mac.c:23` | `raise_with` が `-Wmissing-noreturn` 警告。`__attribute__((noreturn))` 1 行で消える |
| L5 | `Peripheral#services` | `@services` nil 時 `ClosedError` raise (本来は別エラー型が適切 — discover 未呼びと close 後を区別したい) |
| L6 | `CBMCentral.delegatesLock` / `knownPeripherals` | 蓄積のみ、削除無し。`Central#close` 実装時に entry purge + `peripheral.delegate = nil` (`c.queue.sync` 内で) が必須 |

### Phase 2 デプロイ待ち (バグじゃない)

- `test/integration/test_write.rb` / `test_subscribe.rb` / `test_subscribe_ractor.rb` は CoreS3 firmware の NUS RX/TX characteristic (`6e400002-...`, `6e400003-...`) を要求。CoreS3 Phase 2 未デプロイ中は omit する作り (`omit "NUS RX characteristic not present (CoreS3 Phase 2 not deployed yet)."`)

## 5. 次セッションでありがちな bug-fix シナリオ

### A. HW 実機テストで初発覚パターン

```bash
# CoreS3 を rake r2p2:reset で 60s window 開ける
BLE_HW=1 bundle exec rake test 2>&1 | tail -30
```

**起きうる失敗**:
- Permission denied (`tccutil reset Bluetooth` で TCC リセット必要なケース)
- scan で発見できない (CoreS3 advertising window 終了済み、再 reset)
- connect timeout (`state_timeout: 5.0` を伸ばす？)
- discover_services が `unauthorized` を `discovery error` でマスクしてる箇所がある可能性
- read GAP Device Name が空文字列で返る (`bytes.force_encoding("UTF-8")` で空 string になる)

### B. concurrency race (long-running subscribe 中)

L2 が顕在化したら typed lock を `CBMPeripheralDelegate` 12 var に被せるリファクタが必要。テンプレ:
```swift
struct ReadState { var error: Error? = nil; var value: Data? = nil; var charUUID: CBUUID? = nil }
private let readState = OSAllocatedUnfairLock<ReadState>(initialState: ReadState())
```
caller 側はメソッド呼ぶ前に `readState.withLock { $0 = ReadState(...) }` で reset、delegate 側は `readState.withLock { $0.value = ... }` で書く。

### C. `Central#close` 真面目に実装

L6 対応。Swift 側に `func close()` 追加 → `delegatesLock.withLock { for (_, d) in $0 { d.peripheral.delegate = nil }; $0.removeAll() }` + `knownPeripherals.withLock { $0.removeAll() }` + `CBMSubscriptionRegistry.shared.purgeAll(under: self)`。Ruby 側 `Central#close` は今 stub なので native 呼び出しに置き換え。

### D. README に書いてある usage が動かない問題

README は Phase 1/2 usage コードを掲載してる。実機未検証なので、HW テスト走らせて取得値が想定通りか確認、ズレがあれば README 修正。

## 7. Build chain 落とし穴

- Swift signature を追加・変更した直後の `bundle exec rake compile` は `CoreBluetoothMac-Swift.h` を再生成しない (incremental compile の限界)。**Swift signature 変更時は必ず `bundle exec rake clean clobber compile`**
- `bundle exec` プレフィックス必須 (`swift_gem/mkmf` を解決するため)
- LSP (SourceKit / clang) は target 横断インデックス不完全で「Cannot find type 'CBMCentral'」「ruby.h not found」を頻発させるが **すべて false positive**。真の真偽判定は `bundle exec rake compile` のみ
